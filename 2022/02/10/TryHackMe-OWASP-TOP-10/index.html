Injection

常见的注入有：

  • SQL注入：发生在用户控制输入的参数传递给SQL查询时。攻击者可以通过构造SQL查询语句达到SQL注入。
  • 命令注入：发生在用户控制输入的参数传递给系统命令。攻击者能够在应用服务器上执行任意系统命令。

OS Command Injection

当输入的参数在web应用中的服务器端代码(如PHP)对托管机器进行系统调用时，命令注入就发生了。命令注入可以使攻击者任意浏览服务器端的文件，还可以生成一个reverse shell，使攻击者连接进入服务器内网。

Broken Authentication

身份验证和会话管理是现代web应用程序的核心组件。身份验证允许用户通过验证自己的身份来访问web应用程序。最常见的身份验证形式是使用用户名和密码机制。用户将输入这些凭证，服务器将验证它们。如果他们是正确的，服务器就会向用户的浏览器提供一个会话cookie。

常见身份验证失效漏洞：

  • 暴力破解，如果只通过用户名和密码验证，攻击者可以通过枚举的方式进行用户名和密码验证。
  • 弱口令，如password admin等简单的默认账号密码。
  • cookie脆弱性：如果cookie包含可预测的cookie值，攻击者可以构造一个用户cookie登录。

XML External Entity

XML外部实体(XXE)攻击是一种利用XML解析器/数据特性的漏洞。它通常允许攻击者与应用程序本身可以访问的任何后端或外部系统交互，并允许攻击者读取该系统上的文件。它们还可能导致拒绝服务(DoS)攻击，或使用XXE执行服务器端请求伪造(SSRF)，诱导web应用程序向其他应用程序发出请求。XXE甚至可能启用端口扫描并导致远程代码执行。

XXE攻击有两种类型：in-band 和 out-of-band (OOB-XXE)

  • 内部XXE攻击中，攻击者可以收到XXE有效载荷的即时响应。
  • 外部XXE攻击中，web程序不会立即响应。

What is XML?

XML(eXtensible Markup Language)是一种可扩展标记语言，他定义了一组规则，用于以人和机器都可读的格式对文档进行编码。是一种用于存储和传输的标记语言。

语法

每个XML文档大多以XML Prolog开始

<?xml version="1.0" encoding="UTF-8"?>

每个XML必须包含一个根元素：

<?xml version="1.0" encoding="UTF-8"?>
<mail>
    <to>falcon</to>
    <from>feast</from>
    <subject>About XXE</subject>
    <text>Teach about XXE</text>
</mail>

<mail>是该文档的根元素，<to>、<from>、<subject>、<text>是子元素。如果XML文档没有任何根元素，那么它将被认为是错误的或无效的XML文档。XML是大小写敏感的语言。

DTD

DTD代表文档类型定义，DTD定义了结构、合法的元素和属性。用来验证XML的规范。

例：以下为note.dtd内的内容

<!DOCTYPE note [<!ELEMENT note (to,from,heading,body)> <!ELEMENT to (#PCDATA)> <!ELEMENT from (#PCDATA)> <!ELEMENT heading (#PCDATA)> <!ELEMENT body (#PCDATA)>]>

我们可以使用这个DTD来验证某些XML文档的信息，并确保XML文件符合该DTD的规则。

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE note SYSTEM "note.dtd">
<note>
	<to>falcon</to>
	<from>feast</from>
	<heading>hacking</heading>
	<body>XXE attack</body>
</note>
  • !DOCTYPE note：定义了一个名为note的文档根元素
  • !ELEMENT note：定义了note元素必须包含以下元素->”to, from, heading, body”
  • !ELEMENT to：定义了to元素的类型为"PCDATA"
  • !ELEMENT from：定义了from元素的类型为"PCDATA"
  • !ELEMENT heading：定义了heading元素的类型为"PCDATA"
  • !ELEMENT body：定义了body元素的类型为"PCDATA"

Broken Access Control

网站有一些页面是受保护的，不受普通访问者的访问，例如，只有网站的管理员用户才能访问一个页面来管理其他用户。如果网站访问者能够访问受保护的页面/他们没有被授权查看的页面，则访问控制被破坏。

Security Misconfiguraion

安全配置错误与其他漏洞不同，因为他们发生在可被正确配置但没有正确配置的情况下。

安全配置错误主要包括：

  • 云服务器上的权限配置不佳。
  • 启用不必要的功能，如服务、页面、账户或特权
  • 密码未修改的默认用户。
  • 过于详细的报错信息，是攻击者发现更多关于系统的信息。
  • 不使用HTTPS报头，或在HTTP中透露太多细节。

Cross-SIte Scripting

跨站点脚本，也称为XSS，是web应用程序中常见的一种安全漏洞。它是一种注入类型，允许攻击者执行恶意脚本，并在受害者的机器上执行。XSS可能存在Javascript，VBScript，Flash和CSS中。有三个主要类型的XSS：

  1. Stored XSS-存储型XSS：最严重的XSS类型，存储在数据库中，长期存在。
  2. Reflected XSS-反射型XSS：网站将此payload作为用户的响应，需要欺骗用户点击攻击者的payload。
  3. DOM-Based XSS-DOM型XSS：DOM代表文档对象模型，是HTML和XML文档的编程接口。它表示页面，以便程序可以更改文档的结构、样式和内容。网页是一个文档，这个文档可以在浏览器窗口中显示，也可以作为HTML源文件。

Insecure Deserialization

不安全的反序列化就是用恶意代码替换应用程序处理的数据，允许攻击者进行DDoS甚至RCE。这种恶意payload利用了web应用程序使用的合法序列化和反序列化过程。

易受攻击点：

  • 电商平台
  • 论坛
  • 应用程序接口(API)

Object

对象是面向对象编程(OOP)的一个重要元素，它由两部分组成：

  • 状态
  • 行为

DeSerialization

序列化是将编程中使用的对象转换成更简单、兼容的格式，以便在系统或网络之间进行进一步处理或存储的过程。

反序列化与此相反，将序列化的信息转换为复杂的形式——一个应用可以理解的对象。

  • 不安全的反序列化发生在来自不受信任方(攻击者)的数据被执行时，因为没有过滤或输入验证，系统认为是可信的，将会执行恶意payload

Insufficient Loggin and Monitoring

当 web 应用程序设置好后，用户执行的每个操作都应该被记录下来。日志记录非常重要，因为在发生事件时，可以跟踪攻击者的操作。一旦他们的行为被追踪，他们的风险和影响就可以被确定。如果没有日志记录，就不可能知道攻击者在获得访问特定 web 应用程序的权限时执行了哪些操作。

日志中存储的信息包括：

  • HTTP状态码
  • 时间戳
  • 用户名
  • API 端点/页面位置
  • IP地址

攻击行为的常见例子：

  • 多次未经授权的尝试(通常是尝试验证或访问未经授权的资源，例如管理页面)
  • 来自异常IP地址或位置的请求:虽然这可能表明有人试图访问特定用户的帐户，但它也可能误报。
  • 使用自动化工具: 特定的自动化工具可以很容易识别，例如使用 User-Agent 头的值或请求的速度。这可以表明攻击者正在使用自动化工具。
  • 常见有效载荷: 在 web 应用程序中，攻击者常常使用跨网站脚本/服务器(XSS)有效载荷。检测这些有效载荷的使用可以表明有人正在对应用程序进行未经授权/恶意的测试。
